# AI Native Marketing OS 产品架构说明

更新时间：2026-05-20

## 1. 项目定位

Trustwin 君信品牌管理的 AI Native Marketing OS 原型。

当前阶段以「热点雷达 + AI营销知识底座 + 用户权限与积分系统」为核心，但长期目标是构建从 Brief、策略、热点、内容、媒介、Campaign 执行到复盘沉淀的 AI 原生营销操作系统。

这不是爬虫平台，也不是单点热点榜工具。它的本质是把外部热点、内部案例、AI 结构化理解、人工审核和业务系统调用连接起来，形成可持续增长的营销知识飞轮。

## 2. 系统总架构

```text
数据源层
→ 知识生产层
→ AI理解层
→ 业务调用层
→ 权限与积分层
```

### 数据源层

热点数据：

- Rebang
- TopHub
- 第三方热点 API
- Chrome 插件补采
- 手动 CSV / Excel 导入

案例数据：

- 用户上传
- 行业案例采集
- 用户从热点营销建议中收藏
- 手动录入

用户数据：

- 飞书登录
- 飞书头像、姓名、邮箱同步
- 本地角色与权限
- 积分余额
- AI 使用记录
- 话题 AI 建议收藏状态

### 知识生产层

案例从原始材料进入后，需要经过完整的生产流水线：

```text
原始案例输入
→ DeepSeek 结构化
→ 人工审核
→ 标签化
→ RAG / embedding 入库
→ 业务系统调用
→ 用户反馈
→ 持续优化
```

### AI 理解层

AI 不只总结案例，而是提炼可复用营销认知：

- AI 核心洞察
- 为什么有效
- 传播规律
- 情绪机制
- 用户心理
- 传播模型
- 平台打法
- 内容结构模型
- 可复用策略模板
- 风险边界

模型分工：

- DeepSeek / OpenAI-compatible LLM 是“大脑”，负责案例结构化、传播规律提炼、热点营销建议和策略生成。
- Embedding 向量模型是“记忆索引”，负责把案例知识资产转成语义向量，支撑相似案例检索、热点匹配案例，以及未来图文、Campaign、PPT 系统调用。
- 当前后台 `/admin` 已提供 DeepSeek 大模型配置和 Embedding 向量模型配置。Embedding provider 支持 `zhipu`、`xinference_bge_m3`、`openai`、`mock`。

### 业务调用层

当前已接入：

- 热点营销建议
- 每日热点简报
- 我的话题库
- 用户用量统计
- 收藏的热点营销建议

规划接入：

- 图文生产
- Campaign 执行
- PPT 提案
- GEO / 品牌 AI 心智管理
- 客户 Brief 策略拆解

### 权限与积分层

系统通过飞书登录识别用户，通过本地权限控制能力边界。

权限和积分用于控制：

- 谁能浏览知识库
- 谁能上传案例
- 谁能触发 AI 结构化
- 谁能编辑 AI 输出
- 谁能审核入库
- 谁能加入 RAG / 热点策略库
- 谁能调整 DeepSeek 与 Embedding 配置
- 谁能修改用户权限和积分

用户侧导航原则：

- 左侧不再保留“工作台”入口，首页本身即工作台。
- 用户区优先显示“我的话题库”“用量统计”。
- 功能模块区用于进入“AI营销知识”。
- 底部保留“账户设置”，展示飞书头像、姓名、邮箱、角色、积分等信息。

前端还提供登录用户水印：

- 所有通过 `AuthGate` 进入的业务页面都会叠加用户水印。
- 水印仅包含当前用户姓名和当天日期。
- 水印采用普通字重、低密度斜向铺底，兼顾桌面和手机浏览可读性。
- 水印用于降低截图外泄和知识产权损失风险，但不替代后端权限校验、飞书登录和操作审计。

## 3. 当前阶段核心模块

### 热点雷达

负责采集和展示五个平台热点：

- 微博
- 抖音
- 小红书
- 知乎
- B站

核心能力：

- 今日热点
- 平台 Tab
- 跨平台共振
- 上升最快
- 高营销价值
- 高风险话题
- AI 营销建议
- 每日简报
- 政治/公共事务话题不可分析

政治与公共议题原则：

- 系统是品牌营销策划辅助工具，不鼓励对政治、外交、公共治理、执法司法等议题做营销借势。
- 前端会把这类话题标记为“不可分析”并禁用 AI 按钮。
- 后端 `/api/ai/analyze/topic/{id}` 和 `/api/ai/recommend/topic/{id}` 也会执行同样拦截，避免绕过前端。

### AI 营销知识底座

负责把案例转化为 AI 可调用知识资产。

核心页面：

- `/cases-admin`
- `/cases-admin/[id]`

核心能力：

- 原始素材管理
- AI 结构化
- 人工审核
- 标签治理
- 知识价值评分
- RAG 入库
- 相似案例
- 可关联热点
- 可被业务系统调用状态

### 用户权限与积分系统

核心页面：

- `/admin`
- `/topics-library`
- `/usage`
- `/settings`

核心能力：

- 飞书用户识别
- 飞书头像展示
- 角色授权
- 细粒度权限
- 积分调整
- AI 使用记录
- 话题 AI 建议收藏
- DeepSeek 大模型配置
- Embedding 向量模型配置
- 积分消耗趋势

用户侧页面：

- `/topics-library`：展示用户生成过的热点 AI 建议，使用结构化策划卡片，不直接暴露 Markdown 原文；支持“全部 / 已收藏”筛选。
- `/usage`：展示用户分析过的话题、用途、生成时间和案例引用，不直接展示后台 action、模型名或固定积分列。
- `/settings`：展示飞书身份、头像、邮箱、角色、积分和最近访问时间。

## 4. 案例状态与页面阶段映射

| 后端状态 | 页面阶段 | 含义 |
|---|---|---|
| `raw` | 原始案例输入 | 已采集但未结构化 |
| `ai_structured` | AI提炼传播规律 / AI识别情绪机制 | DeepSeek 已输出结构化结果 |
| `human_reviewed` | 人工校正与审核 | 人工已修改但未正式批准 |
| `approved` | 策略标签化 | 已审核，可作为高价值知识资产 |
| `embedded` | 知识入库 | 已完成 RAG / embedding |
| `active` | 被业务系统调用 | 可被热点、图文、Campaign、PPT 调用 |

## 5. 知识资产治理原则

AI 可以提炼，人工必须判断。

关键原则：

- 未审核案例不能直接成为正式知识资产。
- AI 结构化结果必须可人工修正。
- 高价值案例需要人工确认。
- 加入 RAG / embedding 前需要明确风险边界。
- 业务系统调用需要记录调用次数和反馈。
- 所有关键操作需要进入审计日志。
- 所有登录后页面需要显示用户水印，便于截图追溯。

## 6. AI 建议引用透明度

每次热点 AI 营销建议都应该展示：

- 引用了哪些案例
- 每个案例的匹配原因
- 相似度 / 匹配分
- 该案例贡献了哪些策略启发

对应模型：

- `TopicRecommendation`
- `TopicRecommendationCaseRef`

当前产品表现：

- 首页和话题详情页会提示“已调用 AI营销知识库生成建议，引用案例 X 个”。
- 不在用户界面强调底层模型名，避免把用户体验做成开发者日志。
- 如果召回案例为 0，需要视为知识库覆盖不足，而不是隐藏事实。

## 6.1 我的话题库与收藏

“我的话题库”不是后台记录表，而是用户自己的热点策略资产库。

当前能力：

- 展示用户点击 AI 建议后生成的所有话题策划结果。
- 每张卡片展示话题标题、案例引用数量、生成时间和结构化建议预览。
- 支持收藏 / 取消收藏。
- 支持“全部 / 已收藏”筛选。
- 收藏状态保存在后端 `TopicRecommendation.is_favorite`，不是浏览器本地状态。

设计原则：

- 不直接展示大段 Markdown 或开发者格式。
- 不显示底层模型名。
- 每次 AI 建议固定消耗 10 积分，用户页不需要反复强调模型和积分明细。
- 后续可把收藏的 AI 建议转化为案例原始素材，进入 AI 营销知识工作台生产流水线。

这会让用户知道建议不是“凭空生成”，而是基于知识库召回和策略推理。

## 7. 未来业务系统调用接口规划

未来图文生产、Campaign 执行、PPT 提案系统不应直接读数据库，而应通过统一 Knowledge API 调用知识资产。

规划接口：

- `GET /api/knowledge/search`
- `GET /api/knowledge/cases/{id}`
- `GET /api/knowledge/models/{marketing_model}`
- `GET /api/knowledge/nodes/{tag}`
- `POST /api/knowledge/retrieve-for-brief`
- `POST /api/knowledge/retrieve-for-topic`

支持检索维度：

- 按营销模型
- 按情绪机制
- 按行业
- 按平台
- 按策略标签
- 按可复用模板
- 按热点适配度

## 8. 下一阶段 OS 化重点

优先补强四个系统化能力：

1. 向量检索与知识召回
2. AI 建议引用案例透明度
3. 操作审计与知识资产治理
4. 未来业务系统统一 Knowledge API
